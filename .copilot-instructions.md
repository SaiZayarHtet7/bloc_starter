# GitHub Copilot Instructions – Flutter Clean Architecture

You are an expert Flutter architect.  
Follow modern Flutter best practices and produce clean, maintainable, and scalable code.

---

## 🔹 Architecture Principles
- Use **Clean Architecture**
- Follow **SOLID**, **DRY**, and **KISS**
- Prefer composition over inheritance
- Keep layers independent and testable

---

## 🔹 State Management
- Use **Flutter BLoC (Cubit)**
- One Cubit per feature responsibility
- UI must not contain business logic

---

## 🔹 Feature Requirements
### Counter Feature
- Implement a simple `counter_cubit`
- Demonstrate clean separation between layers

---

## 🔹 Network Layer
- Use **Dio** for HTTP requests
- Use **Dartz (Either)** for error handling
- Create a reusable `BaseDioClient`
- Add a sample API call (mock or public API) to demonstrate network handling

---

## 🔹 Dependency Injection
- Use Flutter’s **recommended DI approach**
- Prefer `get_it` with clean registration
- No global singletons outside DI container

---

## 🔹 UI & Presentation
- Separate:
  - Pages
  - Widgets
  - Cubits
- Keep widgets small and reusable
- UI must be declarative and readable

---

## 🔹 Routing
- Use **go_router**
- Centralize all routes in `app_router.dart`

---

## 🔹 Theme Management
- Support:
  - Light Mode
  - Dark Mode
  - System Mode
- Persist theme mode using **SharedPreferences**
- Theme logic must be separated from UI

---

## 🔹 Localization
- Support **multi-language localization**
- Use Flutter’s official localization system
- Ensure scalability for adding new languages

---

## 🔹 Code Quality Expectations
- Clear and meaningful naming
- Avoid unnecessary abstractions
- Prefer readability over cleverness
- All code must be production-ready

---

## 🔹 Output Expectations
When generating code:
- Follow the defined folder structure
- Respect Clean Architecture boundaries
- Write idiomatic Flutter & Dart code
- Keep implementations concise and maintainable
